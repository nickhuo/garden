# ToolLLM: Facilitating LLMs to Master 16000+ Real-world APIs (Qin et al., 2023-07-31, ICLR'24 spotlight)

## Why this matters for the wiki

This is the **scale/breadth pole** of the tool-use benchmark space. Where [[tau-bench]] measures *reliability* on a tiny, hand-annotated, deterministic domain (165 tasks total) and [[BFCL]] measures *single-call correctness* via AST matching, ToolBench measures **generalization across a massive, messy, real-world API surface** (16k+ APIs). All three sit on one axis — see [[Tool-Use Benchmarks - BFCL vs tau-bench vs ToolBench]].

It also predates and frames much of the function-calling discussion in the wiki: it's the 2023 open-source answer to "ChatGPT can use tools and LLaMA can't."

## The framework in one paragraph

ToolLLM = data (**ToolBench**) + model (**ToolLLaMA**) + evaluator (**ToolEval**). ToolBench is built fully automatically with ChatGPT in three stages: (1) collect 16,464 RESTful APIs across 49 categories from RapidAPI Hub; (2) generate diverse instructions in three difficulty tiers; (3) annotate a valid solution path — an ordered sequence of (thought, API call, response) triples — for each instruction using a depth-first search.

## Instruction tiers

- **I1** — single-tool instructions.
- **I2** — intra-category multi-tool (tools from the same RapidAPI category).
- **I3** — intra-collection multi-tool (tools spanning a curated collection).

This is the analogue of BFCL's simple/multiple/parallel split — but graded by *how many real APIs must be composed*, not by call structure.

## DFSDT — the methodological contribution

Linear ReAct/CoT commits to one reasoning trace and gets trapped when an API call fails or returns garbage. **DFSDT** (Depth-First Search-based Decision Tree) lets the model expand and evaluate multiple branches, backtracking from dead ends. ~+13% pass rate over CoT in multi-API settings. Conceptually adjacent to [[Tree of Thoughts]] applied to tool-call trajectories rather than to text reasoning.

## ToolLLaMA + neural API retriever

Feeding 16k APIs into context is impossible, so a **neural API retriever** recommends a relevant subset per instruction (an early, training-time instance of the [[Tool Search Tool]] idea — retrieve tools instead of statically listing them). ToolLLaMA fine-tuned on the annotated paths reaches ChatGPT-comparable performance and generalizes zero-shot to the out-of-distribution APIBench (the [[Gorilla]] benchmark).

## ToolEval — evaluation

LLM-based (ChatGPT) automatic evaluator with two metrics:

- **Pass Rate** — fraction of instructions completed within budget. ToolLLaMA ≈ 50% in DFS multi-API settings; ChatGPT/GPT-4 baseline ≈ 60% on large-scale tasks.
- **Win Rate** — pairwise ChatGPT-judged comparison of two solution paths against a reference.

This is a pure [[LLM-as-Judge]] design — contrast with τ-bench's deterministic database-state reward and BFCL's AST match. It's the most scalable of the three and the least faithful (inherits judge bias).

## Known weaknesses / successors

- **RapidAPI instability** — live APIs go down/change/rate-limit, making scores irreproducible. **StableToolBench** (arXiv:2403.07714, 2024) adds a caching server + simulated API responses to stabilize evaluation.
- ToolEval's ChatGPT judge is a single-model judge; agreement with humans not as rigorously validated as later judge work.
